# ТЗ — Marketplace: AI-поиск услуг/page (Механика 2)

> Дополняет `docs/MARKETPLACE-SEARCH-AND-REQUEST.md` (видение) и реализованную Механику 1
> (`docs/TZ-Marketplace-Tenant-Search.md`, поиск тенанта — закрыта, `hub.company_cache` +
> `search_company_cache` RPC). Эта механика — на уровень глубже: ищем не компанию, а конкретное
> предложение (page/booking_config), как и зафиксировано в видении.

## 1. Чего не хватает (по аналогии с разведкой Механики 1)

`hub.company_cache` денормализует профиль ЦЕЛИКОМ — нет ни одной записи per-page/per-предложение.
Прежде чем писать миграцию — нужна та же разведка, что делалась для Механики 1, но по `pages`:

**Поручить агенту явно проверить, не предполагать:**
1. Сколько у тенанта обычно опубликованных `pages` — одна (типичный случай по `BOOKING-MODEL.md`/
   `ARCHITECTURE.md`) или может быть много? Это определяет, схлопывается ли механика 2 в
   механику 1 для большинства тенантов, или это реально разные результаты поиска.
2. Какие текстовые поля есть на `page`/`page_blocks`, пригодные для индексации: `pages.title`/
   `description` (i18n), содержимое info-блоков, `catalog_items.name`/`description`, заголовки
   секций. Полные `blocks` (JSON-структура) индексировать не нужно — только текстовые куски.
3. Есть ли уже какой-то webhook/sync с уровня page в Hub, или весь обмен сейчас только на уровне
   tenant (`sync-to-hub.ts` шлёт `vitrina_page_slug` — только ОДНУ ссылку, не список).

## 2. Объём

**В этой фазе:** новая денормализованная таблица в Hub на уровне предложения (не тенанта),
с тем же FTS-подходом, что в Механике 1 (`tsvector` + GIN, не embeddings — то же решение,
по той же причине: дёшево, без внешних вызовов на каждый поиск).

**Не входит:** booking-специфичные поля (цена, доступность слотов) в индексе — это бы означало
постоянную ресинхронизацию при каждом изменении доступности, слишком тяжело для поискового кэша.
Поиск находит ПРЕДЛОЖЕНИЕ, актуальную доступность/цену человек видит, уже перейдя на саму
страницу Vitrina (как и сейчас все ссылки из Hub ведут на live-страницу, не статичный снимок).

## 3. Схема (предварительная, уточнить после разведки §1)

```sql
create table hub.listing_cache (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid not null references public.tenants(id) on delete cascade,
  page_slug text not null,
  title jsonb not null default '{}'::jsonb,        -- i18n
  short_text jsonb not null default '{}'::jsonb,    -- агрегированный текст (см. §1.2)
  categories text[] not null default '{}',          -- наследует от company_cache тенанта,
                                                       -- если у page нет своих
  search_vector tsvector,
  synced_at timestamptz not null default now(),
  unique (tenant_id, page_slug)
);
create index listing_cache_search_idx on hub.listing_cache using gin(search_vector);
create index listing_cache_categories_idx on hub.listing_cache using gin(categories);
```

Триггер пересчёта `search_vector` — по аналогии с `company_cache` из Механики 1 (тот же паттерн,
не изобретать новый).

## 4. Sync Vitrina → Hub

Расширить (не заменить) существующий webhook-поток. Сейчас `sync-to-hub.ts` шлёт одну строку на
тенанта с одним `vitrina_page_slug`. Нужен **отдельный** webhook-вызов на уровне страницы (или
расширение текущего payload массивом страниц — решить по факту разведки §1.1: если у тенантов
обычно 1-3 страницы, можно слать списком в одном вызове; если бывает много — отдельный endpoint
с пагинацией). Триггеры синка — те же события, что уже шлют sync сейчас (сохранение/публикация
страницы), плюс публикация/снятие с публикации конкретной page должны обновлять/удалять
соответствующую строку `listing_cache`.

## 5. Поиск и API

`hub.search_listing_cache(p_keywords, p_categories, p_tenant_id?, p_limit)` — RPC по аналогии с
`search_company_cache`. `POST /api/marketplace/search-listings` — тот же AI-парсинг текста в
структурированный фильтр (переиспользовать `parse-marketplace-query.ts` из Механики 1, не писать
вторую копию AI-промпта — расширить, чтобы при необходимости различал «ищут компанию» vs «ищут
конкретную услугу», или просто всегда матчить оба уровня и показывать смешанный результат —
решить на этапе реализации, не предрешать здесь).

## 6. UI

На странице `/marketplace` (уже существует с Механики 1) — результаты теперь могут быть двух
видов: карточка тенанта (как сейчас) и карточка конкретного предложения (заголовок page, переход
сразу на `/p/{slug}`, не на профиль тенанта). Не обязательно два раздельных списка — можно одна
лента с пометкой типа результата, решить по UX-простоте при реализации.

## 7. Тест-план

1. Разведка §1 — пришли реальные цифры/находки до начала миграции, не пропускать.
2. `listing_cache` наполняется при публикации page (не только при первом sync тенанта).
3. FTS находит совпадения по словам из `title`/`short_text`, как в Механике 1.
4. Снятие page с публикации — соответствующая строка `listing_cache` удаляется/помечается
   неактивной (не остаётся висеть и не выдаётся в поиске).
5. AI-парсинг по-прежнему возвращает структурированный фильтр (регресс на Механику 1 — не
   сломать существующий `/api/marketplace/search`).
6. UI показывает оба типа результатов (тенант + предложение) без визуальной путаницы.
