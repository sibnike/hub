# ТЗ — Marketplace: AI-поиск тенанта (Механика 1)

> Дополняет `docs/MARKETPLACE-SEARCH-AND-REQUEST.md` (видение трёх механик) и разведку по
> `company_cache`/каталогу/webhook. Реализуется в **mega-hub** (репозиторий Hub), затрагивает
> также Vitrina (расширение webhook-payload). Самая дешёвая из трёх механик, делаем первой.

## 1. Что показала разведка — на что это влияет

- `hub.company_cache` достаточен по структуре (categories[]/tags[]/country с GIN-индексами уже
  есть), но **недостаточен по содержанию** для семантического поиска: только `short_description`
  (краткий i18n-текст из профиля), нет `city`, нет содержимого страниц, нет full-text/embedding
  индекса вообще.
- Текущий поиск в `/e/{slug}/guide/catalog` (client-side substring по одному событию,
  `GuideCatalogClient`) **не переиспользуется** — другая задача (single-event in-memory поиск
  vs кросс-тенантный backend-поиск). UX-паттерн фильтров (категории, страна) можно взять как
  референс для интерфейса, не для бэкенда.
- В mega-hub нет интеграции с Anthropic API вообще — заводить с нуля (env, serverless route,
  по образцу `lib/ai/call-anthropic.ts` в Vitrina, не изобретать другой подход).
- `meeting_requests` не существует — не нужна для этой механики (это механика 3), но если в
  команде будет молчаливое ожидание, что попутно появится — явно нет, не в этом ТЗ.

## 2. Объём

**В этой фазе:** AI-поиск по тенантам (не по конкретным услугам/страницам — это механика 2,
не входит). Человек вводит свободный текст («нужен гид по горам в Алматы») → получает список
релевантных тенантов с краткой карточкой → переход на существующий профиль (решить, на какую
страницу — см. §5, открытый вопрос).

**Не входит:** поиск по услугам/page (механика 2), формирование запроса (механика 3), сам
маркетплейс-домен/лендинг (где физически живёт `yanbada.com` — отдельный нерешённый вопрос из
видения, не блокирует эту механику, она может временно жить под `/e/marketplace/` в Hub или
аналогичным путём до решения).

## 3. Расширение данных для поиска

### 3.1. Webhook payload (Vitrina → Hub)

Добавить в payload `sync-to-hub.ts`:
- `city` — уже есть в `company_profiles`, просто не передаётся сейчас, дешёвый фикс.
- Опционально (обсудить целесообразность при реализации, не обязательно сразу): расширенный
  текстовый блок — не полные `blocks` страницы (тяжело и не нужно для поиска), а агрегированный
  текст из опубликованных страниц тенанта (заголовки + текстовые info-блоки), если это даст
  значимый прирост релевантности поиска. Если решат не делать — `short_description` остаётся
  единственным источником текста, это ограничение принимается, не блокер.

### 3.2. Поисковый индекс (Hub)

Добавить `hub.company_cache.search_vector tsvector` (Postgres full-text, GIN-индекс) ИЛИ
embedding-колонку (`vector`, через pgvector) — **выбор между FTS и embeddings нужно сделать
явно, не угадывать**:
- FTS — дёшево, без внешних вызовов, но только точные/близкие по словам совпадения («горы»
  не найдёт «альпинизм»).
- Embeddings — семантический поиск (синонимы, смысл), но требует вызова Anthropic/любой
  embedding-модели при каждом обновлении `company_cache` и при каждом поисковом запросе,
  дополнительная инфраструктура и стоимость.

Рекомендация для первой версии: **начать с FTS** (`to_tsvector('russian', name || short_description
|| categories || tags)`) — дёшево, без внешних вызовов, закрывает большинство реальных запросов.
AI-слой (см. §4) добавляется НЕ как замена FTS, а как помощник в формулировке запроса — это
снижает нагрузку на embedding-инфраструктуру в первой версии.

## 4. AI-слой

Не embedding-поиск (см. §3.2, отложено), а **AI как интерпретатор запроса**: пользователь пишет
свободным текстом, Claude разбирает текст в структурированные параметры (категории из
`industry_categories`, ключевые слова, страна/город, если упомянуты) → дальше обычный SQL-запрос
по `company_cache` с FTS + array-фильтрами по categories/tags/country (используя уже существующие
GIN-индексы). AI не отвечает напрямую списком тенантов — только переводит человеческий текст в
структурированный фильтр, сам поиск остаётся детерминированным SQL.

Завести в mega-hub (по образцу Vitrina, не изобретать другой подход):
- `ANTHROPIC_API_KEY` в env
- `lib/ai/call-anthropic.ts` (или аналог)
- Serverless route `/api/marketplace/search` — принимает свободный текст, вызывает AI для
  парсинга, формирует SQL-запрос, возвращает список тенантов

## 5. Открытые вопросы (не наш вывод, явно решить до реализации)

1. Куда ведёт карточка тенанта в результатах поиска — на `/e/{slug}/company/{tenantSlug}` (Hub,
   контекст события) или на `/h/{tenantSlug}` (Vitrina Hub-витрина тенанта)? Текущие профильные
   страницы оба завязаны на контекст (события или собственной витрины) — для маркетплейса нужен
   контекст-нейтральный вариант, возможно новый роут.
2. Где физически живёт сама страница поиска (`/marketplace`, отдельный `event` с `type=marketplace`,
   или будущий корневой `yanbada.com`) — см. `MARKETPLACE-SEARCH-AND-REQUEST.md` §«Где это физически
   живёт». Эта фаза может временно жить под существующим Hub-роутингом, не дожидаясь решения по
   корневому домену, но это нужно подтвердить явно, не предполагать.
3. `city` отсутствует в payload сейчас — добавление тривиально, но решить, нужен ли он для
   первой версии поиска или можно отложить.

## 6. Тест-план

1. Webhook передаёт `city` в `company_cache` (если решили добавить) — сверить на реальном
   тенанте через Supabase MCP, не предполагать.
2. FTS-запрос на тестовых тенантах находит ожидаемые совпадения по словам из `short_description`/
   `categories`/`tags` (не только точное совпадение всей фразы).
3. AI-парсинг свободного текста («нужен гид по горам в Алматы») возвращает структурированный
   фильтр (категория, город), не просто эхо текста.
4. Поиск по структурированному фильтру из п.3 возвращает релевантных тенантов через тот же
   SQL-путь, что и обычный фильтр (не отдельная нестандартная логика).
5. Регресс: существующий каталог `/e/{slug}/guide/catalog` не затронут изменениями (новый
   функционал — отдельный роут, не модификация существующего клиентского поиска).
