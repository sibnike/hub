# ТЗ — Marketplace: формирование запроса (Механика 3)

> Дополняет видение (`docs/MARKETPLACE-SEARCH-AND-REQUEST.md`) и закрытые Механики 1/2 (поиск
> тенанта/услуги). Это ядро маркетплейса: человек пишет свободным текстом, что ему нужно («две
> пары горных лыж»), AI находит подходящих тенантов (через поиск из 1/2), запрос уходит им,
> тенант либо видит готовую цену, либо отвечает своей.
>
> **Объём этого ТЗ — только поток 3a (внешний запрос, посторонний заявитель).** Поток 3b
> (внутренний запрос — тенант ищет исполнителя из своего кабинета, результат встраивается в его
> же систему бронирования) описан в §1b как отдельная, более сложная следующая фаза — не делать
> в рамках этой задачи.

## 1. Архитектурная развилка — где тенант отвечает на запрос (решить ДО реализации)

`hub.meeting_requests` не существует (только заглушка кнопки в H-11) — это поле чистое, но есть
более фундаментальный вопрос: **где тенант видит и обрабатывает входящий маркетплейс-запрос?**

- **Вариант A — новый раздел в Hub** (`/exhibitor/marketplace/requests`). Проблема: весь текущий
  Exhibitor-контур Hub привязан к участию в конкретном событии (`event_participations`) — у
  маркетплейса нет события (открытый вопрос «где физически живёт» из видения ещё не решён).
  Потребует либо нового контура авторизации тенанта в Hub вне событий, либо привязки маркетплейса
  к одному постоянному `hub.events` («маркет» как вечное событие) — тоже нерешённый вопрос.
- **Вариант B — запрос приходит в существующий inbox Vitrina** (`submissions`), как уже приходят
  заявки из Touchin (`docs/INTEGRATION-VITRINA-SUBMISSIONS-FROM-TOUCHIN.md` — внешняя система
  создаёт submission через API, не пишет в БД напрямую). Тенант видит запрос там же, где видит
  обычные заявки — ничего нового не нужно строить на стороне UI Vitrina, только новый источник
  данных. **Соблюдает «золотое правило»** (`YANBADA_ARCHITECTURE.md` §7: Hub никогда не пишет в
  схему `public`) — Hub дёргает API Vitrina, не лезет в её таблицы напрямую, тот же паттерн, что
  уже работает для Touchin.

**Рекомендация (не финальное решение, подтвердить с Николаем перед стартом):** Вариант B. Меньше
нового кода (не нужен новый кабинет/контур авторизации в Hub), тенант видит запрос там, куда уже
смотрит привычно, не нужно решать вопрос «маркет как событие» прямо сейчас — он остаётся открытым
для UI поиска (где человек ИЩЕТ), не блокирует то, куда ОТВЕЧАЕТ тенант.

## 1a. Уточнения от Николая — что добавляется к потоку

- **Проверка занятости на дату.** Прежде чем включать тенанта в список целей запроса (или сразу
  после, как фильтр) — нужно реально проверить его доступность на запрошенную дату через
  существующий публичный API доступности Vitrina (`/api/booking/slots` или `getAvailableDates`/
  `getAvailableStartTimes` — то, что уже работает для V-17…V-22, не строить новую проверку).
  Если AI распознал в запросе конкретную дату — это часть матчинга, не просто текстовая
  релевантность.
- **Уровень адресации — тенант, не сотрудник, в v1.** Сам Николай отметил: по сути это часто
  не тенант целиком, а конкретный сотрудник/исполнитель («гид Иван свободен, гид Петр — нет»).
  Это прямое совпадение с открытым риском №4 из `MARKETPLACE-VISION-FIT.md` §16 («конфликт
  `tenant_staff` vs `performer` — нужен link, не merge», красная зона, не реализовано). Делать
  адресацию на уровне сотрудника в первой версии — серьёзно увеличивает объём (нужна модель
  исполнителя вне контекста одного тенанта). **Решение для v1: адресация на уровне тенанта**
  (как и Механики 1/2), проверка занятости — общая занятость тенанта/его ресурсов на дату, не
  персональная по сотруднику. Адресация по конкретному сотруднику — отдельная будущая итерация,
  явно не входит сюда.
- **Открытая цена с обеих сторон.** Заявитель указывает сумму, на которую согласен (необязательное
  поле в запросе) — тенант видит её. Тенант либо принимает, либо отклоняет, либо предлагает
  другую цену с комментарием. Это уже заложено в `proposed_price`/`response_message` на
  `marketplace_request_targets` (§2) — без изменений схемы, просто подтверждение, что это и есть
  нужный поток.

## 1b. Второй поток — внутренний запрос тенанта (3b, отдельная очередь, не в объёме первой части)

Отличается от внешнего потока (3a, §1-§6 этого документа) принципиально по ИТОГУ, не только по
источнику:

- **3a (внешний, описан выше):** заявитель посторонний человек, итог совпадения — просто связь
  (submission/уведомление), ничего не меняется в системах бронирования ни у кого.
- **3b (внутренний, новое):** заявитель — **сам тенант** из своего кабинета («мне нужен водитель
  12 марта оттуда-то туда-то с машиной» — собирает программу тура). Итог совпадения —
  согласившийся исполнитель (другой тенант/его ресурс) должен **появиться в системе бронирования
  заявителя** как наёмный ресурс на конкретный день — то есть встроиться в `booking_resources`/
  `booking_assignments` (Vitrina) заявителя, не просто прислать уведомление.

### Почему это сложнее технически

Сейчас `booking_assignments` (V-19d/V-24) ссылается только на `tenant_staff` ОДНОГО тенанта —
концепции «внешний тенант как ресурс на день» в схеме нет вообще. Нужно либо:
- новый тип записи в `booking_resources` (`type='external'`) с ссылкой на отвечающего
  `tenant_id` (кросс-тенантная ссылка — раньше такого в модели не было, нужно явно решить, как
  это сочетается с RLS, которое сейчас строго на `tenant_id`),
- либо отдельная сущность поверх (`booking_external_assignments`) — не смешивать с обычными
  `tenant_staff`-назначениями напрямую, безопаснее для существующих гарантий RLS.

### Общее с 3a (переиспользуется)

- Тот же AI-парсинг текста запроса.
- Тот же поисковый слой (Механики 1/2) для поиска подходящих исполнителей.
- Та же проверка занятости на дату (§1a) — здесь даже критичнее: согласие должно ОДНОВременно
  блокировать слот у отвечающего (race-safety, как в V-20 — два разных тура не должны нанять
  одного и того же водителя на один день).
- Та же модель отклика (`marketplace_request_targets`: принять/отклонить/предложить цену).

### Рекомендация по очерёдности

Делать **3a первым** (уже специфицировано, меньше архитектурного риска — нет кросс-тенантной
записи в booking-таблицы). **3b — отдельная следующая фаза**, требует отдельного ТЗ с явным
проектированием кросс-тенантной ссылки и race-safety при подтверждении (переиспользовать паттерн
`pg_advisory_xact_lock` из V-20, не изобретать новый механизм блокировки). Не смешивать в одну
задачу — разный уровень риска и сложности.

## 2. Сущности (Hub — место создания и маршрутизации запроса)

```sql
create table hub.marketplace_requests (
  id uuid primary key default gen_random_uuid(),
  requester_name text not null,
  requester_contact text not null,        -- телефон или email, как свяжутся с заявителем
  request_text text not null,             -- свободный текст «две пары горных лыж»
  ai_parsed jsonb null,                   -- структурированный разбор (категории, ключевые слова,
                                             -- возможно количество/даты — формат уточнить при
                                             -- реализации, переиспользовать парсер из Механики 1
  status text not null default 'open',    -- open | closed
  created_at timestamptz not null default now()
);

create table hub.marketplace_request_targets (
  id uuid primary key default gen_random_uuid(),
  request_id uuid not null references hub.marketplace_requests(id) on delete cascade,
  tenant_id uuid not null references public.tenants(id),
  vitrina_submission_id uuid null,        -- id созданной submission в Vitrina (если Вариант B)
  status text not null default 'sent',    -- sent | viewed | responded | declined | selected
  proposed_price numeric null,            -- если тенант отвечает открытой ценой
  response_message text null,
  responded_at timestamptz null,
  created_at timestamptz not null default now()
);
```

Один запрос → N targets (по числу подходящих тенантов из AI-матчинга через Механику 1/2 поиск) —
это и есть «не одному тенанту, а нескольким сразу», как в видении.

## 3. Поток

1. Человек на странице маркета пишет текст запроса + контакты.
2. AI парсит текст (переиспользовать `parse-marketplace-query.ts` из Механики 1 — расширить под
   формат запроса, не писать новый промпт с нуля).
3. По структурированному фильтру — поиск через `search_company_cache`/`search_listing_cache`
   (Механики 1/2), берём топ-N релевантных тенантов (N — настраиваемый лимит, не присылать всем
   подряд). Если в запросе распознана дата — дополнительно проверить доступность каждого
   кандидата через существующий публичный API Vitrina (§1a), исключить тех, кто на эту дату
   занят полностью (если у тенанта вообще нет booking-конфигов — пропустить проверку, не
   исключать его автоматически только из-за отсутствия календаря).
4. Создаётся `marketplace_requests` + по одной строке `marketplace_request_targets` на каждого
   тенанта.
5. **Если Вариант B (рекомендация):** для каждого target — Hub вызывает API Vitrina (по аналогии
   с Touchin-интеграцией, HMAC-подписанный запрос, не прямая запись в БД) → создаётся submission
   у этого тенанта с данными запроса → `vitrina_submission_id` сохраняется на target → тенант
   видит её в обычном inbox, дальше работает существующий Telegram/email-канал уведомлений
   Vitrina (ничего нового изобретать не нужно — Vitrina уже умеет уведомлять о новой submission).
6. Тенант отвечает (через Vitrina — статус submission, либо комментарий) → нужен **обратный**
   канал Hub ← Vitrina, чтобы заявитель на маркете увидел ответ/цену. Это либо webhook от Vitrina
   к Hub при смене статуса этой конкретной submission (новый, симметричный существующему
   `sync-to-hub`), либо периодический polling — webhook предпочтительнее (дешевле и быстрее),
   решить при реализации, не блокирует начальную часть потока (создание и маршрутизация запроса).

## 4. UI

- Форма запроса на странице маркета (рядом или вместо обычного поиска — решить по UX при
  реализации): текст + имя + контакт → submit.
- Подтверждение заявителю (email/на странице) — что запрос отправлен N тенантам, дальше что
  ожидать (учитывая, что обратный канал ответа — открытый вопрос §3 п.6, на первой итерации
  можно ограничиться «с вами свяжутся напрямую по указанному контакту», не строить сложный
  трекинг статуса для заявителя сразу).

## 5. Тест-план

1. Текст запроса парсится AI в структурированный фильтр (переиспользует парсер Механики 1,
   не падает на простых бытовых формулировках типа «две пары горных лыж»).
2. По фильтру находится корректный список тенантов (через существующий поиск 1/2), создаются
   `marketplace_requests` + N `marketplace_request_targets`.
3. (Если Вариант B) Submission реально создаётся у каждого целевого тенанта в Vitrina, видна в
   его inbox, уведомление (Telegram/email) приходит — реальный HTTP-вызов API Vitrina, не прямая
   запись в БД (соблюдение золотого правила, явно проверить).
4. Запрос без подходящих тенантов (нет совпадений в поиске) — обрабатывается без ошибки, не
   падает (явная ветка, не просто пустой список без объяснения заявителю).
5. Лимит N тенантов на запрос соблюдается — не уходит всем подряд при широком/нечётком тексте.
6. Если в запросе распознана дата — тенант, у которого на эту дату нет свободных слотов (по
   реальной проверке через API доступности), не попадает в список целей; тенант без
   booking-конфигов вообще — не исключается автоматически.

## 6. Открытые вопросы (явно решить, не угадывать)

1. Вариант A vs B (§1) — архитектурное решение, нужно подтверждение перед стартом.
2. Обратный канал статуса/цены к заявителю (§3 п.6) — webhook или другое, можно начать без него
   и добавить отдельной итерацией.
3. Лимит N тенантов на запрос — конкретное число.
4. Формат `ai_parsed` — уточнить при реализации, не специфицирован жёстко здесь.
