# Промпт: Marketplace — формирование запроса (Механика 3)

Контекст: `docs/TZ-Marketplace-Request.md` — прочитать целиком перед началом. Это Hub
(mega-hub), затрагивает Vitrina (создание submission через API, проверка доступности через
существующий публичный booking API). Опирается на Механики 1/2 (поиск тенанта/услуги, закрыты) —
переиспользуй их поисковый слой, не строй параллельный.

## Важно — архитектурное решение пока предварительное

ТЗ §1 рекомендует Вариант B (запрос приходит в обычный `submissions`-inbox Vitrina, как сейчас
работает интеграция с Touchin — внешняя система создаёт submission через API, не пишет в БД
напрямую). Это рабочее предположение, не финально подтверждённое Николаем. **Прежде чем строить
публично видимые части (форма на странице маркета, реальная отправка запросов живым тенантам) —
уточни у Николая, что Вариант B действительно то, что нужно.** Начинать со схемы (§2) и
AI-парсинга (внутренняя логика) можно сразу, это не зависит от решения A/B.

## Не входит в эту задачу

Поток 3b (внутренний запрос — тенант ищет исполнителя из своего кабинета, результат должен
встроиться в его собственный `booking_resources`/`booking_assignments` как наёмный ресурс на
день) — описан в ТЗ §1b как отдельная следующая фаза, явно сложнее (кросс-тенантная ссылка в
booking-схеме, race-safety при подтверждении). НЕ делать сейчас, только поток 3a (внешний
заявитель).

## Задача

1. Hub: миграция `hub.marketplace_requests` + `hub.marketplace_request_targets` (точная DDL —
   ТЗ §2).

2. AI-парсинг текста запроса — переиспользуй `parse-marketplace-query.ts` из Механики 1, расширь
   под формат запроса (не пиши новый промпт с нуля). Если в тексте есть дата — извлеки её в
   структурированный вид (ТЗ §1a).

3. Маршрутизация (ТЗ §3): поиск через `search_company_cache`/`search_listing_cache`, топ-N
   тенантов (лимит — реши при реализации, например 5-10, не присылай всем подряд). Если есть
   дата — для каждого кандидата проверь доступность через **существующий публичный API Vitrina**
   (`/api/booking/availability`/`/api/booking/slots` — найди точное имя в коде Vitrina, не
   угадывай) — исключи тех, кто полностью занят на эту дату. Тенант без booking-конфигов вообще
   — не исключать автоматически (отсутствие календаря ≠ занятость).

4. Уровень адресации — **тенант, не сотрудник**, в этой версии (ТЗ §1a — явное решение, не
   делать адресацию на уровне отдельного исполнителя, это отдельная будущая задача).

5. (Если подтверждён Вариант B) Для каждого target — Hub вызывает API Vitrina (HMAC-подписанный,
   по образцу Touchin-интеграции — найди и переиспользуй существующий паттерн, не изобретай
   новый) → создаёт submission с данными запроса у этого тенанта → сохрани `vitrina_submission_id`
   на target. Дальше уведомление тенанту идёт через уже существующий канал Vitrina (Telegram/
   email при новой submission) — ничего нового не строить здесь.

6. Обратный канал ответа (тенант → заявитель, ТЗ §3 п.6) — можно отложить на отдельную итерацию,
   не блокирует первую часть потока (создание+маршрутизация запроса). Если делаешь — webhook от
   Vitrina к Hub при смене статуса/ответе на конкретную submission, симметрично существующему
   `sync-to-hub`.

7. UI формы запроса на странице маркета — текст + имя + контакт → submit. Подтверждение
   заявителю — простое («запрос отправлен N тенантам, ожидайте связи»), без сложного трекинга
   статуса на первой итерации.

## Тестирование (ТЗ §5, всё обязательно)

- AI парсит текст в структурированный фильтр, не падает на простых формулировках.
- По фильтру создаются `marketplace_requests` + корректное число `marketplace_request_targets`.
- (Если Вариант B подтверждён) Submission реально создаётся у каждого целевого тенанта через
  настоящий HTTP-вызов API Vitrina, не прямую запись в БД — явно проверить (золотое правило:
  Hub никогда не пишет в схему `public` напрямую).
- Запрос без подходящих тенантов — не падает, обрабатывается явной веткой.
- Лимит N тенантов соблюдается.
- Тенант, занятый на распознанную дату, не попадает в цели; тенант без календаря — не исключается.
