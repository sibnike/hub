# Промпт: Marketplace — AI-поиск услуг/page (Механика 2)

Контекст: `docs/TZ-Marketplace-Listing-Search.md` — прочитать целиком перед началом. Это Hub
(mega-hub), затрагивает Vitrina (расширение sync). Опирается на Механику 1 (закрыта,
`hub.company_cache`/`search_company_cache`/`/api/marketplace/search`) — переиспользуй те же
паттерны (FTS+триггер пересчёта вектора, структура AI-парсинга), не изобретай новые с нуля.

## Шаг 0 — обязательная разведка перед миграцией (ТЗ §1)

Не пропускай, не угадывай:
1. Сколько у тенантов обычно опубликованных `pages` (одна типично, или может быть много) —
   проверь по реальным данным через несколько тенантов, не предполагай.
2. Какие текстовые поля есть на `page`/`page_blocks`/`catalog_items`, пригодные для индексации
   (не индексировать сырой JSON блоков целиком).
3. Текущий объём sync на уровне page — `sync-to-hub.ts` шлёт только ОДИН `vitrina_page_slug`
   на тенанта, нет списка страниц вообще.

Пришли результаты разведки до того, как начнёшь писать миграцию — формат таблицы/схемы зависит
от ответа на пункт 1 (один вызов с массивом страниц vs отдельный endpoint с пагинацией).

## Задача

1. Hub: миграция `hub.listing_cache` (ТЗ §3, уточнить по факту разведки) + FTS-индекс +
   триггер пересчёта `search_vector` (тот же паттерн, что в `company_cache` из Механики 1).

2. Vitrina: расширить sync — публикация/снятие с публикации `page` должны создавать/обновлять/
   удалять соответствующую строку `listing_cache` через Hub. Решение по транспорту (расширить
   текущий payload массивом, или отдельный endpoint) — по факту разведки п.1, не предрешено в ТЗ.

3. Hub: RPC `search_listing_cache` (по аналогии с `search_company_cache`) + endpoint
   `POST /api/marketplace/search-listings`. AI-парсинг — переиспользуй
   `lib/marketplace/parse-marketplace-query.ts` из Механики 1, не пиши вторую копию промпта;
   реши при реализации, нужно ли различать «ищут компанию» vs «ищут услугу» или просто матчить
   оба уровня сразу.

4. UI `/marketplace` — результаты двух типов (тенант / конкретное предложение), переход на
   `/p/{slug}` для предложений (не на профиль тенанта). Решение по форме (единая лента с пометкой
   типа, или раздельные списки) — на твоё усмотрение по простоте.

## Тестирование (ТЗ §7, всё обязательно)

- Разведка прислана и учтена в реализации.
- `listing_cache` наполняется при публикации page, не только при первом sync тенанта.
- FTS находит совпадения по словам из заголовка/текста.
- Снятие page с публикации — строка удаляется/деактивируется, не висит в поиске.
- AI-парсинг из Механики 1 не сломан (регресс на существующий `/api/marketplace/search`).
- UI показывает оба типа результатов без путаницы.
