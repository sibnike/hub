# Промпт: Marketplace — AI-поиск тенанта (Механика 1)

Контекст: `docs/TZ-Marketplace-Tenant-Search.md` — прочитать целиком перед началом. Это репозиторий
**mega-hub**, не Vitrina. Затрагивает также Vitrina (webhook payload) — если это разные Cursor-сессии,
часть задачи (§3.1, добавление `city` в payload) может потребовать отдельного промпта в Vitrina,
уточни у Николая, если не уверен, в каком репозитории сейчас работаешь.

## Задача

1. (Опционально, обсудить целесообразность с Николаем перед стартом) Vitrina:
   `lib/company-profile/sync-to-hub.ts` — добавить `city` в payload (уже есть в
   `company_profiles`, просто не передаётся).

2. Hub: миграция — `hub.company_cache.search_vector tsvector` + GIN-индекс (FTS на русском,
   `to_tsvector('russian', name || short_description || categories || tags)`). НЕ embeddings в
   этой версии — см. ТЗ §3.2, явное решение в пользу FTS для первой версии.

3. Hub: завести Anthropic API с нуля (по образцу Vitrina `lib/ai/call-anthropic.ts`, не
   изобретать другой подход) — `ANTHROPIC_API_KEY` в env, клиент-обёртка.

4. Hub: `POST /api/marketplace/search` — принимает свободный текст, AI парсит его в
   структурированный фильтр (категория из `industry_categories`, ключевые слова, страна/город,
   если упомянуты — НЕ просто эхо текста), дальше обычный детерминированный SQL-запрос по
   `company_cache` (FTS + array-фильтры categories/tags/country через существующие GIN-индексы).
   AI не выбирает тенантов напрямую, только переводит текст в фильтр.

5. UI результатов поиска — не специфицировано точно (открытый вопрос §5.1 в ТЗ, куда ведёт
   карточка тенанта — уточни у Николая перед версткой, не угадывай между `/e/{slug}/company/...`
   и `/h/{tenantSlug}`).

## Не делать

- Не трогать существующий клиентский поиск `/e/{slug}/guide/catalog` (`GuideCatalogClient`) —
  отдельная, не связанная логика, оставить как есть.
- Не реализовывать `meeting_requests`/механику 3 — вне объёма этой задачи.
- Не делать embedding-поиск — явно отложено на потом (ТЗ §3.2).

## Тестирование (ТЗ §6, всё обязательно)

- `city` передаётся в `company_cache` (если решили делать) — проверить на реальном тенанте.
- FTS находит совпадения по отдельным словам из `short_description`/`categories`/`tags`, не
  только по точной фразе целиком.
- AI-парсинг возвращает структурированный фильтр на тестовых запросах, не просто повторяет текст.
- Поиск по фильтру из AI идёт тем же SQL-путём, что обычный фильтр (не отдельная ветка логики).
- Регресс: `/e/{slug}/guide/catalog` не изменился.
